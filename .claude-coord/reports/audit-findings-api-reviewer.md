## Cross-Cutting Themes

### 1. Exception Hierarchy Fragmentation (CRITICAL)
The codebase has ~30+ exception classes across 8+ modules, but only about half inherit from `FrameworkException`. Modules like checkpoint, circuit breaker, pricing, OAuth, experimentation, and rollback define standalone exception hierarchies rooted at `Exception`. This makes it impossible to write a single top-level error handler that catches all framework-originated errors. This is the highest-impact contract issue found.

### 2. Protocol/Interface Mismatch (CRITICAL + HIGH)
`ToolRegistryProtocol` defines `get_tool()` which no implementation provides (finding #1). The generic `Registry[T]` Protocol has `get()`/`list_all()`/`count()` but only `ToolRegistry` has adapter methods to match; all other registries ignore the Protocol entirely (finding #7). These Protocols exist in the codebase but don't serve their stated purpose of enabling structural subtyping.

### 3. Backward-Compat Aliases as Functions (MEDIUM)
Two backward-compatibility aliases (`DomainExecutionContext`, `ValidationError`) are implemented as functions returning the target class (findings #12, #19). This breaks `isinstance`, `except`, type annotations, and subclassing -- exactly the patterns consumers would use. Both should be proper class aliases or thin subclasses.

### 4. Pervasive Use of `Dict[str, Any]` at Critical Boundaries (MEDIUM)
`SafetyPolicy.validate()`, `StageExecutor.execute_stage()`, `ObservabilityBackend` methods, and `WorkflowDomainState.data` all use untyped dicts at critical interface boundaries. This pushes type errors from import/compile time to runtime and makes APIs hard to use correctly without reading implementation source.

### 5. Naming Inconsistency Across Registries (LOW-MEDIUM)
Six registry-like classes use six different naming patterns for the same operations (get, list, register). While each is internally consistent, the lack of cross-module convention increases cognitive load for developers working across subsystems.

## Positive Observations

1. **Well-structured primary exception hierarchy** in `src/utils/exceptions.py` with `ErrorCode` enum, context preservation, automatic secret sanitization, and `to_dict()` serialization
2. **Clean ABC contracts** for `ObservabilityBackend`, `CheckpointBackend`, `CacheBackend`, `BaseTool`, `BaseLLM`, `ExecutionEngine`, `CollaborationStrategy`, `SafetyPolicy` -- all with comprehensive docstrings
3. **Safety lazy-loading pattern** in `src/safety/__init__.py` with `__getattr__` + `_LAZY_IMPORTS` dict reduces import time while maintaining discoverability
4. **Comprehensive Pydantic schemas** with field validators, model validators, range constraints, and sensible defaults for all configuration types
5. **Security boundary Protocols** (`PromptInjectionDetectorProtocol`, `OutputSanitizerProtocol`, `LLMRateLimiterProtocol`) cleanly decouple the safety layer from security implementations
6. **Safety exception subclasses** are well-differentiated with clear severity defaults, remediation hints, and `from_violation()` factory methods
7. **Consistent `__all__` usage** across all public modules with categorized export groups and comments
8. **Good deprecation patterns** -- `api_key` field migration, `DomainExecutionContext` warnings, `create_llm_provider` deprecation all show awareness of backward compat. The execution needs refinement (function aliases, factory not updated) but the intent is correct
