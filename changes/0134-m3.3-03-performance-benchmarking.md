## CI/CD Performance Gates

### Workflow Configuration

**File:** `.github/workflows/performance-benchmarks.yml`

**Triggers:**
- Push to main/develop branches
- Pull requests
- Nightly scheduled runs (2 AM UTC)

**Jobs:**

   - Generates flamegraph reports

## Architecture Decisions

### Decision 1: Theoretical vs Empirical Query Reduction Test

**Decision:** Use theoretical calculation for query reduction verification

**Rationale:**
- Database schema complexity (WorkflowExecution has many required fields)
- Theoretical calculation is deterministic and fast
- Real implementation verified in m3.3-02 integration tests
- Benchmark focuses on proving the math, not re-testing ObservabilityBuffer

**Alternatives considered:**
- Empirical test with actual database: More complex, slower, schema-dependent
- Mock-based test: Similar to theoretical but more code

**Trade-offs:**
- Pro: Fast, deterministic, schema-independent
- Con: Doesn't test actual ObservabilityBuffer implementation
- Mitigation: Reference to m3.3-02 tests and change log

### Decision 2: Async Tests Without pytest-benchmark Fixture

**Decision:** Run async verification tests as regular async tests, not benchmarks

**Rationale:**
- pytest-benchmark requires synchronous callable
- asyncio.run() cannot be called from within async context
- Primary goal is verification (assert speedup), not benchmark integration
- Timing measurements are done manually with time.perf_counter()

**Alternatives considered:**
- Wrapper with asyncio.run(): Fails due to event loop conflict
- Synchronous wrapper: Would defeat purpose of testing async
- Custom benchmark fixture: Over-engineering for verification tests

**Trade-offs:**
- Pro: Clean async test code, accurate timing
- Con: Not integrated with pytest-benchmark reporting
- Acceptable: Verification is primary goal, benchmarking is secondary

### Decision 3: CI/CD Performance Gates

**Decision:** Implement GitHub Actions workflow with configurable thresholds

**Rationale:**
- Automated performance regression detection
- Configurable thresholds (10% mean, 15% stddev)
- Baseline comparison prevents gradual degradation
- PR comments provide visibility to reviewers

**Configuration:**
```yaml
--benchmark-compare-fail=mean:10%     # Max 10% slowdown
--benchmark-compare-fail=stddev:15%   # Max 15% variance increase
```

**Alternatives considered:**
- No automated gates: Relies on manual review (error-prone)
- Stricter thresholds (5%): Too sensitive to normal variance
- Per-test thresholds: More complex configuration

**Trade-offs:**
- Pro: Catches regressions early, prevents "death by a thousand cuts"
- Con: May require threshold tuning based on CI environment
- Mitigation: Configurable thresholds, nightly trend tracking
